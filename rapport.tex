\section{Introduction}
% ============================================================

Ce projet a pour objectif de concevoir et déployer une application de gestion de bibliothèque
en suivant une architecture microservices. Chaque fonctionnalité est encapsulée dans un service
indépendant, déployable et scalable de manière autonome.

L'architecture microservices a été choisie pour plusieurs raisons :

\begin{itemize}[leftmargin=*]
  \item \textbf{Séparation des responsabilités} : le catalogue de livres et la gestion des emprunts
        sont deux domaines métier distincts qui évoluent indépendamment ;
  \item \textbf{Résilience} : la panne d'un service n'affecte pas l'autre ;
  \item \textbf{Scalabilité} : chaque service peut être mis à l'échelle indépendamment selon la charge.
\end{itemize}

Objectifs techniques du projet :

\begin{itemize}[leftmargin=*]
  \item développer deux microservices REST en Node.js/Express ;
  \item mettre en place une communication synchrone inter-services via HTTP ;
  \item persister les données avec PostgreSQL (une base par service) ;
  \item conteneuriser chaque service avec Docker et publier les images sur Docker Hub ;
  \item orchestrer l'ensemble avec Kubernetes (Deployments, Services, Ingress, Secrets, RBAC, PVC) ;
  \item proposer une interface web React de démonstration.
\end{itemize}

\section{Architecture générale}
% ============================================================

L'application est décomposée en trois couches principales :

\begin{center}
\begin{tabularx}{\textwidth}{>{\ttfamily}p{4.5cm} X}
\toprule
\normalfont\textbf{Composant} & \textbf{Rôle} \\
\midrule
books-service        & Catalogue des livres -- CRUD + disponibilité \\
user-loans-service   & Gestion des emprunts -- orchestration + communication \\
PostgreSQL (books)   & Persistance du catalogue \\
PostgreSQL (loans)   & Persistance des emprunts \\
frontend             & Interface React/Vite -- catalogue + emprunts \\
k8s/base             & Manifests Kubernetes du catalogue \\
k8s/loans            & Manifests Kubernetes des emprunts (RBAC, Secrets, PVC) \\
docker-compose.yml   & Orchestration locale des services backend (books + loans) \\
\bottomrule
\end{tabularx}
\end{center}

\vspace{0.5cm}

\begin{infobox}
\textbf{Flux de communication :}\\[4pt]
\textbf{1. Consultation du catalogue :}\\
\quad Utilisateur $\rightarrow$ Frontend $\rightarrow$ \texttt{books-service} $\rightarrow$ PostgreSQL (books)\\[3pt]
\textbf{2. Emprunt d'un livre :}\\
\quad Utilisateur $\rightarrow$ Frontend $\rightarrow$ \texttt{user-loans-service} $\rightarrow$ PostgreSQL (loans)\\
\quad \phantom{Utilisateur $\rightarrow$ Frontend $\rightarrow$} $\xrightarrow{\text{HTTP interne}}$ \texttt{books-service} $\rightarrow$ PostgreSQL (books)\\[3pt]
\textbf{3. Retour d'un livre :}\\
\quad Utilisateur $\rightarrow$ Frontend $\rightarrow$ \texttt{user-loans-service} $\xrightarrow{\text{HTTP interne}}$ \texttt{books-service}\\
\quad \phantom{Utilisateur $\rightarrow$ Frontend $\rightarrow$ \texttt{user-loans-service} $\rightarrow$} mise à jour statut + PostgreSQL (books)
\end{infobox}

% ============================================================
\section{Service A : Books Service}
% ============================================================

\texttt{books-service} est développé en Node.js avec Express et PostgreSQL.
Il expose une API REST pour la gestion complète du catalogue de la bibliothèque.

\subsection{Fonctionnalités}

\begin{itemize}[leftmargin=*]
  \item lister tous les livres avec filtres (disponibilité, recherche par titre ou auteur) ;
  \item récupérer un livre par identifiant ;
  \item créer, modifier et supprimer un livre ;
  \item exposer un champ métier \texttt{status} : \texttt{AVAILABLE} ou \texttt{BORROWED} ;
  \item permettre à \texttt{user-loans-service} de modifier la disponibilité via des routes PATCH.
\end{itemize}

\subsection{Routes}

\begin{center}
\begin{tabular}{>{\ttfamily}p{5.5cm} p{7cm}}
\toprule
Route & Description \\
\midrule
GET /books                    & lister tous les livres \\
GET /books?available=true     & filtrer les livres disponibles \\
GET /books?search=...         & rechercher par titre ou auteur \\
GET /books/:id                & récupérer un livre \\
POST /books                   & ajouter un livre \\
PUT /books/:id                & modifier un livre \\
PATCH /books/:id/availability & changer la disponibilité \\
PATCH /books/:id/borrow       & marquer comme emprunté (appel inter-service) \\
PATCH /books/:id/return       & marquer comme disponible (appel inter-service) \\
DELETE /books/:id             & supprimer un livre \\
\bottomrule
\end{tabular}
\end{center}

\subsection{Kubernetes (k8s/base)}

L'ensemble des manifests Kubernetes pour \texttt{books-service} est configuré dans \texttt{k8s/base} :
namespace \texttt{library-app}, ConfigMap, Secret, StatefulSet PostgreSQL,
Deployment, Service et Ingress. Application via Kustomize :

\begin{Verbatim}[breaklines=true,breakanywhere=true]
kubectl apply -k k8s/base
\end{Verbatim}

% ============================================================
\section{Service B : User Loans Service}
% ============================================================

\texttt{user-loans-service} est développé en Node.js avec Express et PostgreSQL.
Il prend en charge la logique métier des emprunts et orchestre la communication avec \texttt{books-service}
pour maintenir la cohérence des données entre les deux services.

\subsection{Fonctionnalités}

\begin{itemize}[leftmargin=*]
  \item créer un emprunt après vérification que le livre existe et est disponible ;
  \item refuser automatiquement un emprunt si le livre est déjà prêté (HTTP 409) ;
  \item lister tous les emprunts actifs et historiques ;
  \item lister les emprunts d'un utilisateur spécifique par son identifiant ;
  \item enregistrer le retour d'un livre et remettre sa disponibilité à jour dans \texttt{books-service} ;
  \item supprimer un emprunt ;
  \item health check via \texttt{GET /health} pour la supervision Kubernetes.
\end{itemize}

\subsection{Routes}

\begin{center}
\begin{tabular}{>{\ttfamily}p{5.5cm} p{7cm}}
\toprule
Route & Description \\
\midrule
GET /health              & vérification de l'état du service \\
GET /loans               & lister tous les emprunts \\
GET /loans/user/:userId  & emprunts d'un utilisateur \\
POST /loans              & créer un emprunt \\
PUT /loans/:id/return    & retourner un livre \\
DELETE /loans/:id        & supprimer un emprunt \\
\bottomrule
\end{tabular}
\end{center}

\subsection{Codes de réponse métier}

\begin{center}
\begin{tabular}{>{\ttfamily}p{2cm} p{9cm}}
\toprule
Code HTTP & Signification \\
\midrule
201 & emprunt créé avec succès \\
200 & opération réussie (retour, liste) \\
404 & livre introuvable dans le catalogue \\
409 & livre déjà emprunté -- conflit métier \\
500 & erreur interne \\
\bottomrule
\end{tabular}
\end{center}

\subsection{Kubernetes (k8s/loans)}

L'ensemble des manifests Kubernetes pour \texttt{user-loans-service} est configuré dans \texttt{k8s/loans} :

\begin{itemize}[leftmargin=*]
  \item \texttt{deployment.yaml} : Deployment Node.js avec injection des variables d'environnement ;
  \item \texttt{statefulset-postgres.yaml} : StatefulSet PostgreSQL pour la haute disponibilité des données ;
  \item \texttt{pvc.yaml} : PersistentVolumeClaim de 1\,Gi -- les données survivent au redémarrage des pods ;
  \item \texttt{service.yaml} : Service ClusterIP pour l'exposition interne dans le cluster ;
  \item \texttt{ingress.yaml} : Ingress nginx qui route \texttt{/loans} vers le service ;
  \item \texttt{secret.yaml} : Secrets Kubernetes pour les credentials PostgreSQL ;
  \item \texttt{rbac.yaml} : contrôle d'accès basé sur les rôles.
\end{itemize}

\subsubsection*{Secrets Kubernetes}

Les credentials de la base de données (hôte, port, nom, utilisateur, mot de passe) sont stockés
dans un \texttt{Secret} Kubernetes encodé en base64. Ils sont ensuite injectés en variables
d'environnement dans le container via \texttt{envFrom}. Aucun secret n'apparaît en clair
dans les fichiers de configuration ou dans le code source.

\subsubsection*{RBAC -- Principe du moindre privilège}

Un contrôle d'accès granulaire est mis en place via trois ressources Kubernetes :

\begin{itemize}[leftmargin=*]
  \item \texttt{ServiceAccount loans-service-account} : identité dédiée au pod, séparée du compte par défaut ;
  \item \texttt{Role} : liste explicite des permissions minimales nécessaires au service ;
  \item \texttt{RoleBinding} : association du Role au ServiceAccount.
\end{itemize}

Ce mécanisme empêche le pod d'accéder à des ressources Kubernetes non autorisées,
conformément aux bonnes pratiques de sécurité en environnement de production.

Application :

\begin{Verbatim}[breaklines=true,breakanywhere=true]
kubectl apply -f k8s/loans/
\end{Verbatim}

\section{Base de données}
% ============================================================

Chaque service dispose de sa propre instance PostgreSQL, déployée comme StatefulSet Kubernetes
avec un PersistentVolumeClaim. Cette isolation garantit l'indépendance des services.

\subsection{Table \texttt{books}}

\begin{Verbatim}[breaklines=true,breakanywhere=true]
CREATE TABLE books (
  id        VARCHAR(100) PRIMARY KEY,
  title     VARCHAR(255) NOT NULL,
  author    VARCHAR(255),
  available BOOLEAN DEFAULT true
);
\end{Verbatim}

\subsection{Table \texttt{loans}}

\begin{Verbatim}[breaklines=true,breakanywhere=true]
CREATE TABLE loans (
  id          SERIAL PRIMARY KEY,
  user_id     VARCHAR(100)  NOT NULL,
  book_id     VARCHAR(100)  NOT NULL,
  loan_date   TIMESTAMP     DEFAULT NOW(),
  return_date TIMESTAMP,
  status      VARCHAR(20)   DEFAULT 'active'
);
\end{Verbatim}

Le champ \texttt{status} prend les valeurs \texttt{active} ou \texttt{returned}.
Le champ \texttt{return\_date} est \texttt{NULL} tant que le livre n'est pas retourné.

\section{Communication inter-services}
% ============================================================

La communication inter-services permet aux deux microservices de rester cohérents
sans partager de base de données.

\texttt{user-loans-service} appelle \texttt{books-service} via la variable d'environnement
\texttt{BOOK\_CATALOG\_URL}, dont la valeur dépend du contexte de déploiement :

\begin{center}
\begin{tabular}{p{3.5cm} p{9cm}}
\toprule
\textbf{Contexte} & \textbf{URL} \\
\midrule
Docker Compose & \texttt{http://books-service:3001} \\
Kubernetes     & \texttt{http://books-service.library-app.svc.cluster.local:3001} \\
\bottomrule
\end{tabular}
\end{center}

En Kubernetes, ce DNS interne est résolu par le \textit{kube-dns} :
le service \texttt{books-service} du namespace \texttt{library-app} est contacté directement,
sans passer par un port externe ni par l'Ingress.

\textbf{Note sur les namespaces :} \texttt{books-service} est déployé dans le namespace
\texttt{library-app} ; \texttt{user-loans-service} est déployé dans le namespace \texttt{default}.
L'URL complète avec le namespace permet la communication cross-namespace.

\subsection{Flux d'un emprunt (\texttt{POST /loans})}

\begin{enumerate}[leftmargin=*]
  \item \texttt{GET /books/:id} --- le service vérifie que le livre existe dans le catalogue ;
  \item si le livre est introuvable, retourne \texttt{404} sans créer d'emprunt ;
  \item si un emprunt actif existe déjà pour ce livre en base (\texttt{status = 'active'}), retourne \texttt{409} (conflit métier) ;
  \item \texttt{PATCH /books/:id/borrow} --- le livre est marqué comme emprunté dans \texttt{books-service} ;
  \item l'emprunt est enregistré en base PostgreSQL avec le statut \texttt{active}.
\end{enumerate}

\subsection{Flux d'un retour (\texttt{PUT /loans/:id/return})}

\begin{enumerate}[leftmargin=*]
  \item l'emprunt est mis à jour : \texttt{status = returned}, \texttt{return\_date = NOW()} ;
  \item \texttt{PATCH /books/:id/return} --- le livre est remis en disponibilité dans \texttt{books-service}.
\end{enumerate}

\begin{infobox}
\textbf{Garantie de cohérence :} si l'appel à \texttt{books-service} échoue, l'emprunt n'est pas
créé (ou le retour n'est pas enregistré). Les deux opérations sont toujours synchronisées.
\end{infobox}

\section{Lancement local avec Docker Compose}
% ============================================================

Un fichier \texttt{docker-compose.yml} permet de lancer les deux microservices
et leurs bases de données respectives en une seule commande :

\begin{Verbatim}[breaklines=true,breakanywhere=true]
docker compose up --build
\end{Verbatim}

Le fichier démarre quatre conteneurs : \texttt{postgres-books} (base \texttt{library}),
\texttt{postgres-loans} (base \texttt{loansdb}), \texttt{books-service} et \texttt{user-loans-service}.
Chaque service attend que sa base soit prête avant de démarrer (\texttt{healthcheck}).

Services exposés en local :

\begin{center}
\begin{tabular}{p{5cm} p{6cm}}
\toprule
Service & URL \\
\midrule
\texttt{books-service}       & \texttt{http://localhost:3001} \\
\texttt{user-loans-service}  & \texttt{http://localhost:3002} \\
\bottomrule
\end{tabular}
\end{center}

Le frontend se lance séparément :

\begin{Verbatim}[breaklines=true,breakanywhere=true]
cd frontend && npm install && npm run dev
\end{Verbatim}

Tests rapides :

\begin{Verbatim}[breaklines=true,breakanywhere=true]
curl http://localhost:3001/books
curl http://localhost:3002/health
curl -X POST http://localhost:3002/loans \
  -H "Content-Type: application/json" \
  -d '{"user_id":"u1","book_id":"book-1"}'
\end{Verbatim}

\section{Frontend React/Vite}
% ============================================================

L'interface utilisateur est réalisée en React/Vite et intègre les deux microservices.

Fonctionnalités de l'interface :

\begin{itemize}[leftmargin=*]
  \item catalogue de livres avec recherche et filtre par disponibilité ;
  \item ajout d'un livre via une modale ;
  \item badges \texttt{Disponible} / \texttt{Déjà prêté} sur chaque fiche ;
  \item bouton \texttt{Emprunter} / \texttt{Retourner} sur chaque livre ;
  \item onglet \texttt{Mes Emprunts} connecté à \texttt{user-loans-service} ;
  \item historique des emprunts par utilisateur ;
  \item formulaire de création d'emprunt guidé.
\end{itemize}

Variables d'environnement :

\begin{Verbatim}[breaklines=true,breakanywhere=true]
VITE_BOOKS_API_URL=http://localhost:3001
VITE_LOANS_API_URL=http://localhost:3002
\end{Verbatim}

\section{Conclusion}
% ============================================================

Ce projet nous a permis de mettre en pratique l'ensemble des compétences abordées en cours :
architecture microservices, conteneurisation, orchestration Kubernetes et développement fullstack.

Les principaux acquis techniques sont :

\begin{itemize}[leftmargin=*]
  \item la conception d'une API REST propre avec gestion des codes d'erreur métier ;
  \item la mise en place d'une communication inter-services fiable via DNS interne Kubernetes ;
  \item la sécurisation des credentials avec les Secrets et le principe du moindre privilège RBAC ;
  \item la garantie de persistance des données avec les PersistentVolumeClaims ;
  \item la maîtrise du cycle complet : code $\rightarrow$ Docker $\rightarrow$ Docker Hub $\rightarrow$ Kubernetes.
\end{itemize}

L'application finale permet de gérer un catalogue de livres, créer des emprunts avec
contrôle de disponibilité, enregistrer les retours et visualiser l'état du système
dans une interface web unifiée -- le tout déployé sur un cluster Kubernetes local.
